# CMCM Proof Strategy

This document summarizes the proof architecture for the compositional memory
consistency model (CMCM) artifact. It is intended as public-facing
documentation for readers who want to understand how the Lean proof is
structured before reading the theorem files.

## Main Proof Path

The release proof path is rooted in:

- `CMCM.lean`
- `CMCM/Herd/Relations.lean`
- `CMCM/Herd/Proof.lean`
- `CMCM/Rf.lean`
- `CMCM/RfTheorem.lean`
- `CMCM/CoTheorem.lean`

The central result proves acyclicity for the relation generated by:

- preserved program order edges (`PPOi`)
- external reads-from edges (`rfe`)
- from-read edges (`fr`)
- coherence-order edges (`co`)

The proof does not add new protocol axioms. It works by unpacking the existing
protocol evidence carried by the RF/CO/FR/PPOi definitions and turning each edge
into an explicit protocol-ordering step.

## Core Events

Each request event is associated with a compound linearization point:

- `compoundLin`: the event used as the compound linearization point
- `CLE`: the cluster-level directory event for the request
- `GLE`: the global-level directory event for the request

The important point is that `compoundLin` events are not usually ordered
directly. Different caches and clusters may have independent local timing. The
proof therefore relates compound linearization events through protocol proxy
events: directory accesses, downgrades, grants, predecessors, successors, CLEs,
and GLEs.

## Temporal Relations

The proof uses four basic temporal relations between events:

- `OrderedBefore`: one event finishes before another starts
- `Encapsulates`: one event interval contains another
- `EncapsulatedBy`: the reverse of `Encapsulates`
- `FinishesBefore`: one event finishes before another finishes

These relations are combined into transitive chains where needed. The strongest
available relation is preferred: `OrderedBefore`, `Encapsulates`, and
`EncapsulatedBy` carry more interval structure than `FinishesBefore`.

## Why Proxy Events Matter

The protocol proof is not just an arithmetic ordering argument over timestamps.
An edge such as RF, CO, or FR has protocol meaning: it is witnessed by concrete
communication between caches and directories. The acyclicity proof mirrors that
communication.

For example, a cross-cluster RF edge may connect the writer's compound
linearization point to the reader's compound linearization point through a chain
like:

```text
writer compoundLin
  -> writer CLE
  -> writer GLE
  -> reader GLE
  -> reader CLE
  -> reader compoundLin
```

The actual Lean constructors carry more detailed evidence than this sketch,
including downgrade events, encapsulation facts, directory-ordering facts, and
request-specific linearization cases. The sketch shows the key idea: the
ordering travels through the protocol's communication structure.

## Three Ordering Levels

The acyclicity argument is organized around a hierarchy of protocol order:

1. Global directory level: cross-cluster communication advances through GLEs.
2. Cluster directory level: same-global-level communication advances through
   CLEs.
3. Event level: same-CLE edges advance through ordinary event ordering.

This hierarchy is represented in the proof by protocol-ordering witnesses such
as `ProtoOBLevel`. Cycles are closed by composing the appropriate ordering
level until it would require an event to be ordered before itself.

## Edge Lifting

The main proof turns every `PPOi`, `rfe`, `fr`, or `co` edge into a structured
step between compound linearization events.

The relevant flow is:

```text
R_hknow edge
  -> protocol-specific edge analysis
  -> CLE/GLE/event ordering evidence
  -> compoundLin proxy chain
  -> acyclicity contradiction for cycles
```

The Lean theorem flow is:

```text
edge_to_proto_forward
cmpLinLinLink_acyclic
cmcm_acyclic_of_hknow_compoundLinOrdering
cmcm_acyclic
cmcm
```

The exact names and helper lemmas live in `CMCM/Herd/Proof.lean`.

## Relationship Between compoundLin and CLE

The relation between a request's compound linearization point and its CLE comes
from `dirAccessOfRequest` and the compound protocol's linearization evidence.
The important cases are:

- `encapDir`: the request accesses the directory directly; the compound
  linearization event is inside the CLE.
- `orderBeforeDir`: a predecessor obtained permissions; the request can
  linearize at the cache event, after the predecessor's directory access.
- `orderAfterDir`: a non-coherent weak request on a dirty line has special
  ordering through a successor directory event; proof obligations involving
  this case are handled by the protocol constraints that make the problematic
  branch vacuous for the release proof path.

These cases explain why a uniform direct order between two `compoundLin` events
is too weak: the proof must remember how each `compoundLin` is connected to its
CLE.

## Important Pitfalls

The proof relies on protocol meaning, not only on type-level manipulation.
Several patterns are important when auditing or extending the artifact:

- Do not add new axioms to finish a proof obligation.
- Do not use `dir_ordered` unless both events are directory events at the same
  relevant address and cluster.
- Do not use `dir_ordered` on the same event to derive a strict self-ordering.
- Do not replace proxy chains with an opaque numeric ranking unless the ranking
  is derived from the named protocol events.
- Read the RF/CO/FR definitions before changing a proof case; their
  constructors define the communication scenario that the proof must mirror.
- Keep RF as the foundation for `rfe`; `rfe` is RF plus the external-cache
  condition.
