1 Overview

This is the “Lean blueprint” 1 accompanying the artifact of “A Formally Verified Foundation for Compositional Heterogeneous Coherence”. This document gives an overview of the mechanized proofs, and the blueprint tool helps synchronize the definitions and proofs in the mechanization with this document. In the following, we will discuss the mechanized proofs and their relation to the claims in the paper.

1.1 Checking the Entire Artifact

We have provided a script to run the entire artifact. You can run it by just typing 

./check-artifact.sh

This will elaborate and check the Lean mechanization and print the axioms of the core main theorems. It will also run the Murphi model checker on the case studies. The output should look similar to:

The HTML table is the same as the table printed in the command line. You may view the HTML table if it’s easier for you.

You can run the Murphi Case Study axiom checks with multiple threads using the –threads flag like so: 

./check-artifact.sh --threads <N_threads>

Just replace <N_threads> with the number of threads you want to use. There’s one axiom that takes about 10 minutes to run, so expect to wait around 20 minutes for the model checking to run, if only using 1 thread. For that reason, using at least 2 threads is ideal, as the other thread can complete the other axioms while the other is stuck on the longer axiom.

You can also test the Lean Mechanization Check by adding something that depends on a sorried statement, e.g. 

theorem one_plus_one_false : 1 + 1 = 1 := by sorry
theorem false_for_testing : False := 
  False.elim (Eq.mp (OfNat.ofNat_ne_one._simp_1 2) one_plus_one_false)
#print axioms false_for_testing

Which results in something like 

 ✗ false_for_testing (contains sorryAx)

The rest of this chapter explains the individual parts of the artifact in more detail (what the above script runs), and the subsequent chapters explain the mechanization in even more detail.

1.2 Evaluating the Lean Mechanization

This section discusses how to check and evaluate theorems/lemmas in the Lean mechanization. In VSCode, theorems and lemmas that are verified by lean have blue checkmarks to their left, which means they type check. This docker image should already have the dependencies for VSCode and Lean. However, that the proof type checks does not mean that it is not missing some helper lemma that was axiomatized. To check this, you can run a command (in a separate line) to print all axioms assumed to prove a theorem. For example, for our main theorem we have 

#print axioms CompoundProtocol.enforce_compound_consistency

You can see the message it prints on the right in the infoview panel in VSCode, or if you run it on the command line using 

lake lean CompositionalProtocolProof/CompositionalMCM.lean

it will print it to stdout). The message should read: 

'CompoundProtocol.enforce_compound_consistency' depends on axioms:
  [propext, Classical.choice, Quot.sound]

This means that our main theorem, CompoundProtocol.enforce_compound_consistency only depends on the built-in axioms of propositional extensionality, soundness of quotients and the axiom of choice (our proof is classical). Notably, the axiom sorryAx is missing, since no helper lemma has been admitted using the sorry keyword.

The next Chapter (Chapter 2) provides an overview of the structure of the proof from the paper. You can use it to get an idea of how to navigate parts of the Lean mechanization.

1.3 Running the Murphi Case Study Models

Section 7 in our paper lists two case studies for using combinations of CXL and RCC-O in a compound protocol. We’ve included a script to run the axiom checks on the CXL and RCC-O murphi models for you. You can run them one at a time, or launch multiple threads at a time, by using the threads parameter. The check-artifact.sh script will run the Lean Mechanization check as well as the axiom checks. As mentioned earlier, without passing the thread count to check-artifact.sh, it will run the axiom checks sequentially.

If you’d like to run the axiom model checking separately, you can run the script directly, invoking the python script like so: 

python3 model-check-axioms/scripts/run_axioms.py --clean
  --axioms-file axioms-to-run.txt

The –clean flag deletes old generated files from previous runs. The –axioms-file flag indicates an input file to the script, that contains the list of Murphi axiom files to model check. The axioms-to-run.txt file contains the axioms checked in the case studies.

You can supply the –threads <N> flag to specify how many threads to use. The script uses a default of 50MB for each model checking instance, which is sufficient for all axiom model checking runs (including the longer 10 min axiom).

1.3.1 Use Leanblueprint

This pdf was generated with LaTeX and leanblueprint. To generate a webpage version that includes a graph visualization of the main theorem, and some of the main lemmas and definitions, run the command: 

  $ leanblueprint web

To view the webpage, you will need to serve the webpage first with the command: 

  $ leanblueprint serve

Which will then produce some output: 

  $ leanblueprint serve
  Serving http://0.0.0.0:8000/ 
  Press Ctrl-c to interrupt.

Where you can view the generated lean theorem dependency graph from the locally served URL (the “http://0.0.0.0:8000/” link after “Serving”).

You can generate another copy of this pdf if you desire, with the command: 

  $ leanblueprint pdf

The pdf can be viewed by opening it with a pdf viewing program.
